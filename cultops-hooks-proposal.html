Problems → Hooks

Problem 1: AI sessions mutate production data without approval.
March 27 — two sessions made conflicting harvest_date fixes 37 min apart. $50k estimated impact. No gate prevented either write.

Hook 1: Production DB Mutation Gate

Implement Now

Prevents the #1 most expensive failure mode

gate-production-mutations.sh
PreToolUse → execute_sql on production DB

Intercepts every SQL call to the production Supabase (fonreynkfeqywshijqpi). READ operations pass through. Any INSERT, UPDATE, DELETE, TRUNCATE, DROP, or ALTER is blocked and escalated to you for approval.

#!/bin/bash
# .claude/hooks/gate-production-mutations.sh
#
# Blocks destructive SQL on production DB.
# Context DB (uayyhluztelnfxfvdhyt) and staging (cbxwippkzeszvxewhebd) pass through.

INPUT=$(cat)

# Extract project_id and query from the MCP tool input
PROJECT_ID=$(echo "$INPUT" | jq -r '.tool_input.project_id // empty')
QUERY=$(echo "$INPUT" | jq -r '.tool_input.query // empty')

# Only gate the production database
PRODUCTION_DB="fonreynkfeqywshijqpi"
if [ "$PROJECT_ID" != "$PRODUCTION_DB" ]; then
  exit 0  # Not production — allow
fi

# Normalize query to uppercase for matching
QUERY_UPPER=$(echo "$QUERY" | tr '[:lower:]' '[:upper:]')

# Allow pure reads
if echo "$QUERY_UPPER" | grep -qE '^\s*(SELECT|WITH|EXPLAIN)\b'; then
  # But block SELECT ... INTO and SELECT ... FOR UPDATE
  if echo "$QUERY_UPPER" | grep -qE '\bINTO\b|\bFOR\s+UPDATE\b'; then
    echo "BLOCKED: SELECT INTO / FOR UPDATE on production requires approval." >&2
    echo "Query: $QUERY" >&2
    exit 2
  fi
  exit 0  # Pure read — allow
fi

# Block everything else
echo "BLOCKED: Destructive SQL on production DB ($PRODUCTION_DB)." >&2
echo "Query: $(echo "$QUERY" | head -c 200)" >&2
echo "" >&2
echo "To proceed: re-run with explicit approval or execute manually." >&2
exit 2

Settings config

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "mcp__775d8087-127d-4963-bd11-38f8c3427741__execute_sql",
        "hooks": [
          {
            "type": "command",
            "command": "\"$CLAUDE_PROJECT_DIR\"/.claude/hooks/gate-production-mutations.sh"
          }
        ]
      }
    ]
  }
}

What happens: Claude tries to run an UPDATE on production → hook fires → script detects it's destructive → exits with code 2 → Claude sees the block message and tells you what it wanted to do → you decide whether to approve or not.

Hook 2: Session Context Enforcer

Implement Now

Prevents lazy implementations without context

enforce-context-load.sh
SessionStart → startup | resume

On every session start or resume, injects a mandatory context reminder into Claude's prompt. This isn't a suggestion the AI can skip — it's injected as system context via stdout.

#!/bin/bash
# .claude/hooks/enforce-context-load.sh
#
# Injects mandatory context loading instructions on session start.
# Output goes to stdout = injected into Claude's context.

cat <<'CONTEXT'
=== MANDATORY SESSION PROTOCOL ===

Before taking ANY action in this session, you MUST:

1. Run: SELECT prepare_session_context('<your focus>')
   on project_id: uayyhluztelnfxfvdhyt (context DB)

2. READ the full result before writing any code or SQL.

3. If touching production DB (fonreynkfeqywshijqpi):
   - State what you plan to change and why
   - Wait for explicit user approval before any mutation
   - Production writes are gated by PreToolUse hook

4. If touching code:
   - Read the relevant files FIRST
   - Run npm run build after changes
   - Do NOT create new files without checking if existing ones can be extended

Skipping these steps has caused $50k+ in data corruption.
This is not optional.

=== END PROTOCOL ===
CONTEXT

exit 0

Settings config

{
  "hooks": {
    "SessionStart": [
      {
        "matcher": "startup|resume",
        "hooks": [
          {
            "type": "command",
            "command": "\"$CLAUDE_PROJECT_DIR\"/.claude/hooks/enforce-context-load.sh"
          }
        ]
      }
    ]
  }
}

Hook 3: Post-Edit Build Check

Implement Next

Catches broken builds before they get committed

Agent-based build verifier
Stop → (fires when Claude finishes responding)

Uses an agent-based hook — spawns a subagent that can actually run commands. After Claude finishes a coding task, the agent runs npm run build and checks for TypeScript errors. If the build fails, Claude is told to keep working.

{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          {
            "type": "agent",
            "prompt": "Check if any TypeScript or React source files were modified in this session. If so, run 'npm run build' in the project root and verify it passes. If the build fails, respond with {\"ok\": false, \"reason\": \"Build failed: [first error message]\"}. If no source files were changed or the build passes, respond with {\"ok\": true}.",
            "timeout": 120
          }
        ]
      }
    ]
  }
}

What happens: Claude finishes writing code → agent hook spawns → runs the build → if it fails, Claude automatically continues fixing. You don't have to catch build failures manually.

Hook 4: Context Reinject on Compaction

Implement Next

Prevents context loss in long sessions

reinject-on-compact.sh
SessionStart → compact

When Claude's context gets compacted (summarized to free space), critical rules get lost. This hook re-injects the essentials after every compaction.

#!/bin/bash
# .claude/hooks/reinject-on-compact.sh
#
# Re-injects critical context after compaction.

cat <<'CONTEXT'
=== POST-COMPACTION CONTEXT RELOAD ===

Critical rules that survive compaction:

DATABASES:
- Production: fonreynkfeqywshijqpi (cultops.io, main branch)
- Staging: cbxwippkzeszvxewhebd (staging.cultops.io, dev branch)
- Context DB: uayyhluztelnfxfvdhyt (session memory only)
- NEVER mix these. ALL changes go to BOTH prod and staging.

INVENTORY:
- Never directly update quantities. Use inventoryMovementService.
- batch_id is never null.
- Finalization = Creation. Movement records are audit-only.

DEPLOYMENT:
- Cherry-pick between main and dev. Never merge branches.
- Push to dev triggers staging deploy. Push to main triggers prod.

PRODUCTION DB WRITES ARE GATED:
- The PreToolUse hook blocks all mutations on fonreynkfeqywshijqpi
- State what you want to change, wait for user approval

=== END RELOAD ===
CONTEXT

exit 0

Settings config

      {

          {
            "type": "command",
            "command": "\"$CLAUDE_PROJECT_DIR\"/.claude/hooks/reinject-on-compact.sh"

Hook 5: Session Activity Logger

Nice to Have
log-tool-use.sh
PostToolUse → all tools

Logs every tool call to a local file. Useful for post-session review and debugging conflicting changes across parallel sessions.

#!/bin/bash
# .claude/hooks/log-tool-use.sh
INPUT=$(cat)
TOOL=$(echo "$INPUT" | jq -r '.tool_name')
SESSION=$(echo "$INPUT" | jq -r '.session_id')
TIMESTAMP=$(date -u +"%Y-%m-%dT%H:%M:%SZ")

# For SQL tools, capture the query
QUERY=$(echo "$INPUT" | jq -r '.tool_input.query // .tool_input.command // "n/a"' | head -c 200)

echo "$TIMESTAMP | $SESSION | $TOOL | $QUERY" >> ~/.claude/cultops-audit.log
exit 0

What This Does and Doesn't Solve

Problem Solved? Notes
AI mutates production data unsupervised Blocked Hook 1 gates all production writes. You approve or deny.
AI skips context loading Injected Hook 2 injects protocol on every session start. Can't be skipped.
Broken builds get committed Blocked Hook 3 auto-runs build check and forces Claude to fix failures.
Context lost during compaction Injected Hook 4 re-injects critical rules after every compaction.
Parallel sessions conflicting Partially Audit log (Hook 5) helps diagnose. Full prevention needs Paperclip-style coordination.
Staging forgotten after prod change No Could add a Stop hook that checks "did you apply to both DBs?" — future enhancement.

Scope & Applicability

Claude Code & Cowork: These hooks apply directly. Both run on the Claude Code runtime, so .claude/settings.json governs behavior.

AntiGravity: Separate tool, separate context. These hooks won't apply there unless AntiGravity supports Claude Code hooks (needs verification). For AntiGravity sessions, the production DB gate would need to be implemented at the database level instead — e.g., a Postgres function that checks a session lock table before allowing writes.

Paperclip (future): If you adopt Paperclip later, these hooks become the safety layer underneath it. Paperclip handles coordination and delegation; hooks handle the hard blocks that no orchestrator should be able to override.
